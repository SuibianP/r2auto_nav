Maze Solving Overview

  • Maze is modelled as a nonsimple weighted undirected graph
  • Modified wall follower algorithm
  • Due to the limited number of nodes, can use brute force (exhaustion) with no time/space complexity considerations

"Move forward with purpose."

  • Follow the wall when there is no different edges to choose
  • Attempt to stay in the middle between the two walls by constantly adjusting the direction

scan Pattern

  • A sudden jump to smaller values in other directions

Target Projection Overview

  • Use OpenCV for IR image identification
  • Signal maze solving node for control priority once target is confirmed

Target Identification

  1. Do a binary threshold on the grayscale input image
  2. Apply morphological operations to eliminate noise
  3. Detect if there is a large enough spot to be regarded as the target. If so, signal for control.

Target Identification

  1. Traverse the current edge to determine the point where IR is strongest
  2. Make a full turnaround at the point to find the best orientation to the target

Shooting

If there is a wall blocking the target, it will blindly shoot the balls at different angles according to the intensity of the bright spot.
Otherwise it will try to detect the distance of the target using LiDAR as a reference.

Freeze Protection

One bash script spawns the actual working node.

    If
  • There is no callback operation within threshold
  • The node encounters either too many exceptions within a given duration/an unrecoverable state (SIGSEGV, SIGBUS, etc.)

the script will terminate the process with SIGKILL and restart

Timeout

Every single non-trivial operation must have a timeout value after which it will be abandoned.

Miscellaneous

Power Consumption

Possibly use a minimal ROS-supported OS image. Unused services/processes/devices(LAN/HDMI/Bluetooth/LED/audio)/etc. should be deactivated. Strictly no GUI apps/DE/WM. Our repo wiki

Computationally intensive jobs will be transferred to another higher performance computer to process via ssh tunnel.

Even to the extent of doing everything on a remote machine, with the onboard SBC only serving as a Internet device for OpenCR.

CPU, I/O governors, CPU clock and cores can be tweaked by experiment to be at the most power efficient setting without disrupting service.

Programming Languages
